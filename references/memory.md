# Memory System — 跨会话记忆

> Worker 跨会话记忆持久化系统。每个 Worker 角色有专属的记忆文件，用于在多次会话间积累和复用知识。

---

## 设计原则

- **角色隔离**: 每个 Worker 写自己的记忆，不跨角色改写
- **追加式更新**: 新发现追加到对应章节，不覆盖历史
- **PM 只读汇总**: PM 启动新 Worker 前读取对应记忆文件，注入到 Worker 上下文
- **定期清理**: 每个记忆文件标注"知识半衰期"，过期条目由 PM 在后续会话清理

---

## 目录结构

```
memory/
├── README.md                        # 本说明
├── roles/                           # 各 Worker 角色的知识积累
│   ├── architect.memory.md          # 技术调研缓存、架构约束、已知限制
│   ├── ba.memory.md                 # 领域术语表、Supervisor 偏好
│   ├── ux.memory.md                 # 设计决策、组件库约定
│   ├── coder.memory.md              # 代码约定积累、踩坑记录
│   └── tester.memory.md             # 测试模式、常见回归点
├── session/
│   └── current-state.md             # 当前会话状态 (谁在等什么、阻塞项)
└── decisions/                       # 架构决策记录 (ADR)
    └── <ADR-0001>-<title>.md
```

## 记忆写入规则

每个 Worker 完成任务后，在写 `docs/pm/from-<role>/<date>-done.md` 的同时，追加更新自己的记忆文件:

```markdown
## <YYYY-MM-DD> — <本次任务简述>

### 新发现 (Discovered)
- 发现了什么约束/限制
- 学到了什么模式
- 什么假设被证伪了

### 可复用 (Reusable)
- 本次产出的可复用部分 (调研结论/代码片段/测试用例)
- 什么模式可以套用到类似场景

### 需要关注 (Watch)
- 潜在风险、未完事项
- 下次需要确认的问题
- 即将过期的假设 (如"这个 API 下个版本会变")
```

## PM 读取时机

| 时机 | 读取内容 | 用途 |
|------|---------|------|
| **派发 Worker 前** | 对应 `memory/roles/<role>.memory.md` 最近 3 条 | 注入 Worker prompt 作为参考上下文 |
| **新会话启动** | `memory/session/current-state.md` | 恢复上下文: 上次进行到哪、谁在等什么 |
| **做同类需求** | 搜索关键词匹配的历史记忆条目 | 复用过去的调研结论和设计决策 |

## 跨会话恢复流程

新会话启动时 (或 PM 调用 `/crewkit:resume` 时)，PM 执行:

```
1. 读取 memory/session/current-state.md
2. 解析"活跃需求"表 → 了解哪些需求在进行中
3. 解析"阻塞项" → 了解什么卡住了
4. 解析"最近完成" → 了解上次做了什么
5. 确定"下一检查点" → 第一个动作是什么
6. 向 Supervisor 汇报: "上次我们做到 X，现在 Y 在等 Z。下一步是..."
```

---

## 记忆清理规则

PM 在每次会话结束时检查:

| 清理条件 | 动作 |
|---------|------|
| 条目超过 90 天 | 归档到文件底部 `## 归档 (<年份>)` 章节 |
| 已被证伪的假设 | 追加一行 `~~已证伪: <原因>~~` 而非删除 |
| 同一主题 ≥5 条 | 合并为一条总结 + 引用历史条目 |
| memory 文件 > 500 行 | 归档最早条目，保持活跃条目 < 200 行 |

## current-state.md 更新规则

PM 在以下事件发生时立即更新 `memory/session/current-state.md`:

| 事件 | 更新内容 |
|------|---------|
| Worker dispatch | 将需求加到"活跃需求"表，标注当前阶段和等待谁 |
| Worker 完成 | 更新对应阶段状态，移动至"最近完成" |
| 发生阻塞 | 加入"阻塞项"表，标注阻塞原因和所需行动 |
| 验收通过 | 从"活跃需求"移除，更新"下一检查点" |
| Supervisor 打回 | 标注打回原因和重新进入的阶段 |

## ADR 写入规则

当 Architect 做出重大技术决策时 (如选择技术栈、决定架构模式、引入新依赖)，PM 应要求创建 ADR:

```markdown
# ADR-0001: <决策标题>

> 日期: YYYY-MM-DD | 状态: 提议 / 已采纳 / 已废弃 | 决策人: Architect

## 背景
<为什么需要做这个决策>

## 决策
<我们决定做什么>

## 备选方案
| 方案 | 优点 | 缺点 | 为何未选 |
|------|------|------|---------|
| A | — | — | — |
| B | — | — | — |

## 后果
<采纳后的影响: 好处、风险、需要的后续行动>
```

---

## ADR 自动创建触发条件

PM 在以下决策点自动创建 ADR (填入模板 → 标记"提议"):

| 触发场景 | ADR 标题示例 | 决策人 |
|---------|-------------|--------|
| Architect 推荐技术方案 (≥2 候选) | "选择 X 作为 Y 模块的技术方案" | Architect |
| 引入新的第三方依赖 | "引入 X 库用于 Y 功能" | Architect |
| 架构模式变更 | "从 X 模式迁移到 Y 模式" | Architect |
| Supervisor 在两个方案间拍板 | "Supervisor 选定方案 B" | Supervisor |
| Architect 判定"不可行"后选择替代路径 | "X 不可行，采用替代方案 Y" | Architect + PM |

ADR 文件命名: `ADR-<0001起递增>-<kebab-case标题>.md`

## Worker 耗时追踪

PM 在状态更新时记录耗时:

```
每个 Worker 完成时:
  - 记录: 角色、开始时间、结束时间、耗时分钟数
  - 写入 current-state.md 对应需求行

/crewkit:status 展示:
  - 每个 Worker 的实际耗时 (括号内)
  - 每个需求的累计耗时 (⏱️ 标记)
  - 阻塞时长 (如有)

目的: 帮助 Supervisor 了解时间分配，发现瓶颈 Worker。
```
