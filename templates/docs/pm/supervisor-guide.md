# Supervisor 操作指南

> 你是整个流程的"甲方"——只需要对 PM 说话，其余角色由 PM 调度。

---

## 你需要做什么

### 三个动作

| 动作 | 时机 | 输出 |
|------|------|------|
| **提出需求** | 任何时候 | 自然语言描述想做什么 |
| **确认需求文档** | PM 提交 BA 产出的需求文档时 | PASS / 打回 + 具体原因 |
| **最终验收** | PM 提交测试报告时 | PASS / 打回 + 具体原因 |

中间所有阶段 (Architect/UX/Coder/Tester) 都不需要你介入——PM 会消化这些角色的产出，用你能理解的方式汇报。

### 不需要你做的

- 不需要看代码
- 不需要操心"谁在干什么"
- 不需要跟踪进度（PM 会汇报）

---

## 怎么给出有效的反馈

### 打回需求文档时

坏的反馈: "这个需求不对" (PM 不知道怎么改)
好的反馈:

```
需求文档有 3 个问题:
1. 用户故事 US-2 缺少"是什么价值"——为什么用户需要这个？
2. 验收标准 AC-3 太模糊，"正常显示"怎么定义？
3. 遗漏了一个边界场景——用户已有数据再新建时怎么处理？
```

**原则**: 每个打回点附带"少了什么"或"具体哪不对"。

### 打回最终验收时

坏的反馈: "界面不太对" (Coder 和 UX 不知道问题在哪)
好的反馈:

```
有 2 个问题:
1. 列表页在空数据时只显示"暂无数据"，但没给"新建"入口——用户不知道接下来该干嘛
2. 提交按钮在网络慢时没有 Loading 状态，点了没反应，用户体验差
```

## 监督者决策清单

PM 可能在以下情况需要你来决策:

| PM 给你的问题 | 你需要说什么 |
|-------------|-----------|
| "方案 A 和方案 B，选哪个？" | 选一个，或者提你的偏好条件 |
| "这个功能技术上不可行，替代方案是 X 或 Y" | 选一个，或者调整需求范围 |
| "两个角色对同一个设计有分歧" | 给出你的判断 |
| "进度被 X 阻塞了" | 决定等待还是降级范围 |

## 你与 PM 的典型交互

```
你: "我想做一个用户反馈收集功能"
  ↓
PM: "好的，我拉 BA 做需求澄清。先确认下: 是运营后台能看到反馈就行，还是需要用户端入口？"
  ↓
你: "两个都要"
  ↓
PM: "明白。这是 M 级需求，预计 2-3 个会话完成。我先拉 BA 出文档。"
  ↓
... PM 消失了，后台在工作 ...
  ↓
PM: "BA 需求文档好了，你确认下: docs/ba/user-feedback.md"
  ↓
你: "看了。第 3 条验收标准改成'支持按时间筛选'，其他 OK"
  ↓
PM: "收到。接下来拉 Architect 和 UX 并行设计。"
  ↓
... PM 消失了 ...
  ↓
PM: "全部完成了，测试报告在这里: docs/tester/user-feedback.md。测试通过，可以验收。"
  ↓
你: "OK，我用一下看看"
  ↓
你: "有个问题——反馈列表没分页，数据多了会很慢"
  ↓
PM: "收到，我拉 Coder 补分页。(这是 S 级，一个会话内改完)"
```

---

## 关键原则

1. **只对 PM 说话** — 不需要知道内部有几个 Worker 在跑
2. **给具体反馈** — "少了 X"比"不对"有用 10 倍
3. **信任中间流程** — Architect/UX/Coder/Tester 之间的协作交给 PM
4. **看不懂就问** — PM 的角色是"技术翻译官"，你应该能理解他的汇报
